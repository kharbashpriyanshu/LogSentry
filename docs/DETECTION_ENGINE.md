# Detection Engine & Rule Framework

The Detection Engine is the core security analysis component of LogSentry. It consumes normalized `LogEvent` objects generated by the Parsing Engine, evaluates them against a suite of specialized rules, and produces standardized `DetectionAlert` objects when malicious activity is identified.

## Architecture

The engine strictly adheres to **Clean Architecture** and **SOLID Principles**:
- **Strategy Pattern**: Each detection rule is an independent strategy implementing the `BaseRule` interface.
- **Registry Pattern**: The `RuleRegistry` manages the discovery and instantiation of rules, allowing for dynamic loading without modifying engine logic (Open/Closed Principle).
- **Single Responsibility Principle**: Rules only concern themselves with logic to identify a specific attack vector. They do not parse logs, route alerts, or store data.

## Detection Pipeline
```text
Raw Log ──► Parser ──► LogEvent ──► DetectionEngine ──► Rule ──► DetectionAlert
```

## Rule Interface (`BaseRule`)
Every rule must implement:
- `rule_name`, `rule_version`, `description`, `severity` (Properties)
- `match(event: LogEvent) -> bool`: The logic to detect the attack.
- `generate_alert(event: LogEvent) -> DetectionAlert`: The constructor for the standard alert format.

## Standardized `DetectionAlert` Schema
Alerts are heavily structured to facilitate future integration with SIEM dashboards, AI SOC Analysts, and Threat Intelligence platforms.
Key fields include:
- `alert_id` (UUID)
- `rule_name`, `severity`, `confidence`, `risk_score`
- `attack_type`, `mitre_technique`, `mitre_tactic`
- `evidence`, `raw_log_reference`

## Supported Detections (Sprint 2)
1. **SQL Injection**: Detects common SQLi payloads (e.g., `UNION SELECT`, `OR 1=1`).
2. **Cross-Site Scripting (XSS)**: Detects XSS vectors (e.g., `<script>`, `onerror=`).
3. **Path Traversal**: Detects attempts to access arbitrary files (e.g., `../`, `/etc/passwd`).
4. **Command Injection**: Detects OS command injections (e.g., `;`, `&&`, `$( )`).
5. **Directory Enumeration**: Detects access to sensitive paths (e.g., `/admin`, `/.git`).
6. **Brute Force**: Stateful detection of multiple failed authentication attempts within a time window (Thresholds configurable via `.env`).

## Error Handling
The `DetectionEngine` isolates rule execution in a `try/except` block. A bug in a custom rule will never crash the ingestion pipeline; it will simply be logged, and evaluation will continue for remaining rules.
